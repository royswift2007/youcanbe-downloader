# 阶段0执行清单

## 1. 阶段定位

阶段 0 的目标不是新增用户可感知的大功能，而是先把当前程序的“可维护性、可回归性、可诊断性”建立起来。

没有阶段 0，后续阶段 1 到阶段 4 的任何增强都会变成高风险试错。

本阶段的核心任务只有四类：

- 固化当前基线
- 增强日志与错误分类
- 建立最小自动化测试骨架
- 形成可重复执行的冒烟清单

---

## 2. 本阶段要解决的问题

当前项目存在以下现实问题：

1. 下载命令构建完成后，缺少足够清晰的结构化复现信息。
2. 下载失败时，错误日志与错误类别没有完全收口。
3. 历史记录虽然已存在，但缺少足够清晰的失败阶段标记。
4. 仓库目前没有 `tests/` 目录，后续一旦改 builder 或 history，容易回归。
5. 目前适合先把“如何确认没有改坏”做出来，再继续往上叠功能。

---

## 3. 本阶段完成标准

只有满足以下全部条件，阶段 0 才算完成：

- 已存在最小测试目录
- 至少为 `ytdlp_builder` 建立最小测试
- 至少为 `history_repo` 建立最小测试
- 下载前能输出结构化命令摘要
- 失败能区分阶段
- 已形成当前主流程的冒烟清单
- 冒烟至少走通一次

---

## 4. 本阶段不做什么

本阶段明确不做以下内容：

- 不新增下载功能
- 不新增 Generic 模式
- 不新增 FFmpeg 工作台
- 不新增 Deno Hook
- 不大改 UI 布局
- 不修改下载主逻辑的业务策略
- 不改历史仓库为通用多任务结构

如果开发人员在阶段 0 中做了上述内容，视为偏离阶段目标。

## 8. 详细开发步骤

### 8.1 建立最小测试骨架

目标：

让后续开发至少可以跑最基本的自动化验证。

步骤：

1. 新建 `tests/` 目录。
2. 选择项目当前最容易接入的测试运行方式。
   建议优先使用 `pytest`。
3. 先保证测试可以被发现和执行，不要求一开始就覆盖很多逻辑。
4. 建立最小测试约定：
   - builder 测试放 `tests/test_ytdlp_builder.py`
   - history 测试放 `tests/test_history_repo.py`
   - 输入校验测试放 `tests/test_input_validators.py`

注意：

- 这个阶段的目标不是高覆盖率。
- 是为了给后续阶段建立最小保护网。

### 8.2 为 ytdlp_builder 建立最小测试

目标：

确保核心命令拼装在后续阶段可回归。

最少要覆盖：

1. 默认输出模板拼装
2. 自定义文件名输出模板拼装
3. 音频模式参数拼装
4. 视频模式参数拼装
5. 元数据与封面相关参数
6. 限速参数
7. 节流参数
8. PO Token 打开时的 extractor args

建议测试方式：

- 构造一个最小 `task` 对象
- 调用 `build_ytdlp_command`
- 断言命令列表中是否包含预期参数

不要做的事：

- 不在此阶段测试真实下载
- 不把外部程序执行耦合进 builder 单测

### 8.3 为 history_repo 建立最小测试

目标：

确保历史写入与兼容逻辑不会被后续改坏。

最少要覆盖：

1. sqlite 初始化成功
2. 插入成功记录
3. 插入失败记录
4. 旧字段兼容逻辑
5. `video_id` 与 `playlist_id` 提取逻辑

建议做法：

- 使用临时 sqlite 文件
- 使用最小假任务对象
- 测试写入后再读取或断言返回值

### 8.4 增加结构化命令摘要

目标：

让日志具备“看到后可以复现”的能力。

实现要求：

1. 下载开始前记录命令摘要。
2. 摘要中至少包含：
   - 任务 ID
   - URL
   - preset_key
   - format
   - 输出目录
   - 是否音频模式
   - 是否启用 cookies
   - 是否启用 PO Token
   - 关键附加参数
3. 不要求把所有参数毫无筛选地原样打印给普通用户。
4. 但日志中必须能帮助开发人员复盘。

建议实现位置：

- [`core/download_manager.py`](../core/download_manager.py)

### 8.5 统一失败阶段分类

目标：

让失败不再只是“失败了”，而是知道失败发生在哪个阶段。

建议最少分类：

- `title_fetch`
- `command_build`
- `download`
- `postprocess`
- `history_write`

实施要求：

1. 下载管理器在不同位置捕获异常时补上失败阶段。
2. 历史仓库写入时保存该信息。
3. 日志与运行时问题提示尽量复用这套分类。

### 8.6 输出人工冒烟清单

目标：

把当前主流程固化成一份后续阶段都会反复执行的清单。

至少包含：

1. 单视频最佳兼容下载
2. 单视频音频下载
3. 自定义文件名下载
4. 批量解析播放列表
5. 批量入队下载
6. Cookies 失效提示
7. 历史写入成功
8. 下载失败时的日志可读性

## 9. 建议提交顺序

建议按以下顺序提交，而不是一次性全部改完：

1. 提交 1：测试骨架
2. 提交 2：builder 测试
3. 提交 3：history 测试
4. 提交 4：结构化命令摘要
5. 提交 5：失败阶段分类
6. 提交 6：冒烟文档

## 10. 阶段验收清单

验收人检查以下内容：

- `tests/` 是否真实存在
- 测试是否可以执行
- builder 测试是否覆盖关键命令拼装
- history 测试是否覆盖成功与失败记录
- 下载前日志是否有结构化摘要
- 失败时是否能区分阶段
- 当前 YouTube 主流程是否仍然可用
