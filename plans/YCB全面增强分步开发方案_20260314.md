# YCB 全面增强分步开发方案

## 1. 文档目的

本文档用于指导当前 [`YCB.pyw`](../YCB.pyw) 项目的后续增强开发，目标是把现有以 YouTube 为核心的下载器，逐步升级为基于 **yt-dlp + FFmpeg + Deno** 的下载与媒体处理应用。

本文档以 [`YTSage_Analysis.md`](../YTSage_Analysis.md) 为优先参考输入，并结合当前代码现状整理为可直接实施的开发路线、阶段清单、代码落点、验收标准与回滚策略。

修订说明：

- 本文档保留中长期目标，但已明确区分“当前可执行路线”和“中长期路线”
- 当前代码结构下，不应把 Generic、FFmpeg 工作台、Deno 插件体系同时作为短期开发主线

### 2.1 来自 YTSage_Analysis 的优先补全项

`YTSage_Analysis.md` 中提到的能力，后续开发中按以下优先级吸收，且默认优先级高于一般体验优化项。

#### P0：直接影响成功率、下载可控性、核心体验

- 完整字幕系统
  - 手工字幕、自动字幕、两者都要
  - 多语言与正则匹配
  - 字幕格式 `srt/vtt/ass`
  - 外挂、内嵌、两者并存
- SponsorBlock 集成
  - 支持类别选择
  - 默认关闭，避免误裁剪
- 区段下载
  - `--download-sections`
  - FFmpeg fallback
- 代理支持与网络诊断
  - `--proxy`
  - 超时、DNS、认证失败、地区限制、403 分类提示
- 高级参数透传
  - 允许覆盖剩余 yt-dlp 能力
  - 必须带冲突检测

#### P1：信息透明、专业能力、效率提升

- 格式信息表格化
  - 分辨率、编码、协议、体积、HDR、是否需合并
- 元数据与附件导出补齐
  - 缩略图、描述、info.json、章节
  - 区分嵌入与导出
- Browser Cookies 导入
  - `--cookies-from-browser`
- Generic Mode
  - 从只做 YouTube 专业模式扩展到 yt-dlp 支持站点的通用入口
- 剪贴板监听
  - 自动识别 URL
  - 去重与开关控制

#### P2：产品化与运维能力

- 组件中心
  - yt-dlp / ffmpeg / deno 版本显示、更新、校验
- 多语言界面
  - 先做资源字典，不直接上复杂 i18n 体系
- 打包与离线分发
  - Windows 为第一目标

### 2.2 本文档对 YTSage_Analysis 的吸收策略

为避免后续执行时再次出现“分析文档存在但实施文档未明确对应关系”，后续阶段默认遵循以下规则：

1. 凡是 `YTSage_Analysis.md` 已明确提到、且当前项目尚未完整支持的能力，优先进入阶段 1 到阶段 4。
2. 凡是会显著扩展站点覆盖、媒体处理能力或自动化能力的项，进入阶段 5 到阶段 7。
3. 任何新增能力，都必须同时回答三个问题：
   - 对应哪个 yt-dlp 或 FFmpeg 官方能力
   - 放在 UI、预设还是高级透传层
   - 对现有 YouTube 主流程的默认影响是什么

### 2.3 当前程序适配边界

基于当前代码现状，以下内容适合立即进入开发：

- 阶段 0：基线稳固、日志、错误分类、最小回归
- 阶段 1：区段下载、SponsorBlock、代理、高级参数透传、元数据附件补齐
- 阶段 2：字幕系统专业化
- 阶段 3：格式表格化与剪贴板监听
- 阶段 4：认证中心与成功率提升

以下内容应视为中长期路线，不适合在当前结构上直接并行铺开：

- Generic 通用模式
- 独立 FFmpeg 媒体工作台
- Deno Hook 插件体系
- 大规模组件中心、多语言与产品化收尾

原因：

- 当前 URL 识别、任务模型、调度器、历史仓库都明显是 YouTube 强绑定
- 当前仓库还缺少自动化测试基础
- 现阶段最合适的是先把 YouTube 主流程做稳、做深、做可诊断

## 2A. 目标升级：从增强下载器到“yt-dlp + FFmpeg 超级工作台”

### 2A.1 最终产品形态

后续目标不再只是“补几个下载开关”，而是形成三条并行产品线：

1. YouTube 专业模式
   面向当前主用户路径，强调成功率、认证、批量与归档。

2. Generic 通用模式
   面向 yt-dlp 支持的其他站点，强调通用入口、低假设、少站点特化。

3. FFmpeg 媒体处理工作台
   面向本地媒体处理，强调转封装、转码、滤镜、剪辑、拼接与批处理。

### 2A.2 对“把 yt-dlp/ffmpeg 所有功能加到程序里”的实施口径

开发目标可以定义为：

- 在产品层面尽可能覆盖 yt-dlp 与 FFmpeg 官方能力
- 但不把每一个命令行参数都机械做成单独按钮

具体落地为四层：

1. 精选 UI 层
   高频能力做成可视控件。

2. 预设模板层
   把常见工作流做成“策略”或“模板”。

3. 高级参数透传层
   覆盖剩余大部分 yt-dlp 和 FFmpeg 官方参数。

4. Deno 脚本与 Hook 层
   覆盖复杂编排、规则化流程和第三方集成。

换言之，最终目标不是“每个 flag 一个按钮”，而是“用户通过 UI + 模板 + 透传 + 脚本，能调用接近全部官方能力”。

但对当前项目而言，这个目标的正确执行顺序是：

1. 先强化 YouTube 专业模式
2. 再完成中层抽象与去 YouTube 假设
3. 最后进入 Generic、FFmpeg 工作台与 Deno 插件体系

## 3. 总体设计原则

### 3.1 产品分层

后续产品能力按四层设计：

1. 基础领域层  
   `YouTubeDownloadProfile`、任务模型、批量来源模型、历史记录模型。

2. 引擎适配层  
   yt-dlp 命令构建、FFmpeg 后处理编排、Deno Hook 执行器。

3. UI 预设层  
   面向普通用户的精选能力，覆盖高频场景。

4. 高级扩展层  
   附加 yt-dlp 参数、FFmpeg 模板、Deno 插件脚本。

### 3.2 功能覆盖策略

不追求把 yt-dlp 和 FFmpeg 的所有参数做成独立控件，采用：

- 常用能力做成 UI
- 中高级能力做成可持久化预设
- 剩余能力通过“高级参数透传”与脚本扩展覆盖

### 3.2A yt-dlp 官方能力覆盖清单

后续应按能力域而不是零散参数推进。yt-dlp 侧至少覆盖以下能力域：

#### 下载与提取

- 单视频
- 播放列表
- 频道 / Uploads
- Live / 回放
- Shorts
- Generic 站点
- 站点自动识别
- 平铺索引与批量入队

#### 格式与输出

- 自动最佳格式
- 手工格式选择
- 音视频分离流合并
- 纯音频提取
- 强制转封装
- 强制重编码
- 输出模板
- 文件名模板
- 归档文件
- 下载路径与分类输出

#### 字幕、章节、附件、元数据

- 手工字幕 / 自动字幕
- 多语言与正则过滤
- 内嵌字幕 / 外挂字幕
- 字幕格式选择
- 章节嵌入 / 导出
- 缩略图下载 / 嵌入
- 描述文件
- info.json
- 元数据嵌入
- 评论导出、播放列表元数据等进阶导出能力

#### 网络与可靠性

- 代理
- 限速
- 重试
- 节流
- 断点续传
- 并发控制
- 请求头、User-Agent、Referer、额外 extractor 参数
- 失败诊断与建议

#### 认证与风控

- Cookies 文件
- Cookies from browser
- PO Token
- 用户可控 extractor args
- 受限内容提示

#### 后处理与增强

- SponsorBlock
- download-sections
- 音频抽取
- 视频重编码
- 章节处理
- 字幕处理
- postprocessor args

#### 维护与系统能力

- 组件更新
- 版本检测
- 剪贴板监听
- 日志导出
- 配置导入导出

### 3.2B FFmpeg 官方能力覆盖清单

FFmpeg 侧不应只作为 yt-dlp 的隐式后处理器，而应单独形成媒体工作台。至少覆盖以下能力域：

#### 容器与编码

- 转封装
- 视频转码
- 音频转码
- 编码器选择
- 码率、采样率、声道、像素格式等核心参数

#### 基础剪辑

- 起止时间裁剪
- 多段剪辑
- 拼接
- 分割
- 批量裁剪

#### 画面处理

- 缩放
- 裁切
- 旋转
- 帧率变换
- 去隔行
- 加水印
- 简单滤镜链

#### 音频处理

- 音量归一化
- 静音片段处理
- 多音轨处理
- 音视频替换
- 提取背景音 / 保留单轨等常见模板

#### 字幕处理

- 外挂字幕生成
- 字幕烧录
- 字幕内封
- 字幕轨管理

#### 颜色与高级处理

- HDR/SDR 基础转换预设
- 色彩空间相关模板
- 高质量转码模板

#### 媒体信息与诊断

- ffprobe 风格的媒体信息展示
- 编码、时长、流信息、章节、字幕轨显示

### 3.2C 全覆盖目标的执行约束

为了让“超级全面”变成可实现目标，而不是持续膨胀的需求池，后续文档执行时必须遵守：

1. 每个官方能力先判断归属：
   - 核心 UI
   - 高级 UI
   - 模板
   - 透传
   - 脚本扩展
2. 核心 UI 只保留高频、稳定、易理解项。
3. 低频但重要的官方能力优先通过透传或模板覆盖。
4. 如果一个能力只有命令行专家才会直接使用，就不强制做成主界面控件。

### 3.3 实施原则

- 先补齐基础能力和可观测性，再叠加高级功能
- 每阶段都只围绕少数明确目标，不并行展开过多主题
- 每阶段结束必须完成一次真实链路冒烟
- 新增功能默认保持关闭或保守，避免破坏现有下载主流程
- 在完成中层抽象前，不并行推进 Generic、FFmpeg 工作台、Deno 三条主线
- 当前阶段默认以 “YouTube 专业模式” 为主干，其他能力以后置规划为主

## 4. 先做的架构收口

在进入功能阶段前，先完成以下统一动作，否则后续功能会继续分散在 UI 和 builder 中：

### 4.1 扩展下载配置模型

优先扩展 [`core/youtube_models.py`](../core/youtube_models.py) 中的 `YouTubeDownloadProfile`，新增但不限于：

- `subtitle_mode`: `none/manual/auto/both`
- `subtitle_langs`
- `subtitle_format`
- `embed_subs`
- `write_subs`
- `sponsorblock_enabled`
- `sponsorblock_categories`
- `download_sections`
- `proxy_url`
- `advanced_args`
- `cookies_mode`: `file/browser/none`
- `cookies_browser`
- `ffmpeg_profile`
- `ffmpeg_extra_args`
- `hook_profile`
- `generic_mode_enabled`
- `site_mode`
- `request_headers`
- `referer_url`
- `user_agent`
- `archive_file_path`
- `comment_options`
- `playlist_options`
- `live_options`
- `extractor_args`
- `postprocessor_args`

要求：

- 字段命名统一，避免 UI 变量名直接渗透到命令构建
- 给每个新增字段提供默认值，保证旧任务兼容
- 将“单视频”和“批量”统一复用同一 Profile 结构

修订约束：

- 第一轮不要把 Generic、FFmpeg、本地媒体处理、Hook、复杂请求头等字段全部塞进 `YouTubeDownloadProfile`
- 第一轮只扩展与当前 YouTube 下载主流程直接相关的字段：
  - 字幕
  - SponsorBlock
  - 区段下载
  - 代理
  - Browser Cookies
  - 高级参数透传
  - 元数据与附件导出
- Generic 配置、FFmpeg 任务配置、Hook 配置应在后续拆成独立模型，而不是继续膨胀当前 Profile

### 4.2 提取 builder 子模块

建议将 [`core/ytdlp_builder.py`](../core/ytdlp_builder.py) 从单函数拆成以下职责：

- `build_base_command`
- `apply_auth_args`
- `apply_subtitle_args`
- `apply_postprocess_args`
- `apply_network_args`
- `apply_output_args`
- `apply_metadata_args`
- `apply_section_args`
- `apply_playlist_args`
- `apply_extractor_args`
- `apply_advanced_args`
- `validate_advanced_args`

目标：

- 降低后续继续在一个函数里堆参数的风险
- 为每个功能点建立单测入口
- 允许未来在 builder 上下文中接入 Deno/FFmpeg Hook

修订约束：

- 第一轮 builder 拆分只覆盖当前确定要落地的能力
- 不提前为了远期 Generic、FFmpeg、Deno 扩展而引入过多空接口

### 4.3 统一设置持久化入口

当前窗口位置状态在 [`core/settings.py`](../core/settings.py) 中维护。后续新增设置建议拆分为独立的应用配置仓储，例如：

- `config/app_config.json`
- 或基于现有配置仓储继续扩展

最低要求：

- 代理、字幕默认值、高级参数、Deno Hook、组件路径、浏览器 cookies 来源可持久化
- UI 层不直接写散落文件

### 4.4 新增 FFmpeg 工作台配置模型

建议新增独立媒体处理模型，避免将 FFmpeg 本地处理能力继续硬塞进下载 Profile：

- `MediaProcessProfile`
- `MediaInputItem`
- `MediaTaskRecord`
- `FfmpegTemplate`

最低字段建议：

- 输入路径
- 输出路径
- 模板类型
- 视频编码参数
- 音频编码参数
- 过滤器链
- 字幕处理方式
- 起止时间
- 高级参数

### 4.5 新增引擎能力注册表

为了长期管理 yt-dlp / FFmpeg / Deno 三类能力，建议新增“能力注册表”或“功能注册清单”，用于回答以下问题：

- 该能力是否已实现
- 实现在 UI、模板还是透传层
- 位于哪个模块
- 是否已验收

建议新文件：

- `plans/能力覆盖清单_yt_dlp_ffmpeg.md`

该文件用于长期追踪“全能力接入”的完成度，避免只在主方案中写目标，不跟踪完成情况。

### 4.6 自动化验证前置

当前仓库尚未形成最小测试基线，因此本方案修订后要求把自动化验证前置到阶段 0。

阶段 0 至少补以下测试：

- `core/ytdlp_builder.py` 的命令拼装测试
- `core/history_repo.py` 的历史写入与兼容性测试
- 输入校验与 profile 映射测试

在没有这层保障前，不建议开始跨域重构。

## 5. 阶段路线图

## 阶段 0：基线稳固与可回归验证

### 目标

先把“增强前的基线”固定下来，避免后续功能越来越多但无法判断谁破坏了主流程。

### 交付物

- 一份最小回归清单
- 结构化命令日志
- 结构化错误摘要
- 当前能力基线说明
- 最小自动化测试骨架

### 主要改动点

- [`core/download_manager.py`](../core/download_manager.py)
- [`core/ytdlp_builder.py`](../core/ytdlp_builder.py)
- [`YCB.pyw`](../YCB.pyw)

### 分步实施

1. 为下载前的命令构建结果增加结构化日志输出。
2. 区分“标题获取失败”和“正式下载失败”的错误类别。
3. 为队列任务记录最近一次 `command_preview`、`error_summary`、`error_detail`。
4. 为运行时问题增加统一错误码或错误分类。
5. 为 `ytdlp_builder` 和 `history_repo` 建立最小单测。
6. 梳理最小冒烟链路：解析标题、加入队列、开始下载、完成写历史。

### 验收标准

- 单视频下载成功链路可完整记录关键信息
- 批量页入队后仍能正常并发调度
- 失败任务在 UI 或日志中能看出失败类别
- 关键 builder 行为可通过自动化测试复验

### 回滚要求

- 日志增强不改变现有命令参数
- 不影响旧任务对象读写

## 阶段 1：下载核心增强

本阶段直接吸收 [`YTSage_Analysis.md`](../YTSage_Analysis.md) 中价值最高、最影响成功率和可控性的内容。

### 1A. 区段下载

#### 目标

支持 `--download-sections`，并为不适配场景准备 FFmpeg fallback。

#### 代码落点

- [`core/youtube_models.py`](../core/youtube_models.py)
- [`core/ytdlp_builder.py`](../core/ytdlp_builder.py)
- [`ui/pages/single_video.py`](../ui/pages/single_video.py)
- [`ui/pages/batch_source.py`](../ui/pages/batch_source.py)

#### 实施步骤

1. 在 Profile 增加 `download_sections` 字段，格式先限定为单段或多段字符串。
2. UI 增加“开始时间/结束时间”输入，先支持 `HH:MM:SS`。
3. builder 中注入 `--download-sections`。
4. 对非法时间格式做输入校验。
5. 下载失败且命中特定错误模式时，记录“可考虑走 FFmpeg fallback”的诊断，但第一版先不自动切换。
6. 第二小步再实现 FFmpeg fallback，避免第一版复杂度过高。

#### 验收

- 日志中能看到 `--download-sections`
- 输出文件时长接近期望区间
- 非法时间输入不会进入队列

### 1B. SponsorBlock 集成

#### 目标

支持 `--sponsorblock-remove`

#### 实施步骤

1. Profile 增加开关和类别列表。
2. UI 提供默认关闭的复选框。
3. 高级区域允许填写类别，例如 `sponsor,selfpromo,interaction`。
4. builder 注入 SponsorBlock 参数。
5. 在任务摘要中明确显示“启用片段删除”。

#### 验收

- 命令中出现对应 sponsorblock 参数
- 未启用时行为与现在一致

### 1C. 代理支持与网络诊断

#### 目标

显式支持 `--proxy`，并把网络问题转化为可操作提示。

#### 实施步骤

1. 增加全局代理配置与任务级覆盖配置。
2. builder 注入 `--proxy`。
3. 在 [`core/youtube_metadata.py`](../core/youtube_metadata.py) 与下载失败日志中补充代理错误识别。
4. 为超时、403、连接拒绝、DNS、代理认证失败建立诊断文案。
5. 设置页或高级设置卡片加入代理输入框与测试按钮。

#### 验收

- 代理地址可保存并复用
- 常见网络错误有明确提示

### 1D. 高级参数透传

#### 目标

为剩余 yt-dlp 能力提供受控扩展口。

#### 实施步骤

1. UI 增加“高级参数”文本框。
2. 新增参数解析器，支持按空格或按行解析。
3. 建立黑名单，至少禁止：
   - `-o`
   - `--output`
   - `--ffmpeg-location`
   - `--cookies`
   - `--cookies-from-browser`
   - `--exec`
   - `--config-locations`
4. 将冲突项在 UI 中给出阻断提示。
5. 日志中显示“用户附加参数”。

#### 验收

- 合法附加参数可进入命令
- 冲突参数被阻止并说明原因

### 1E. 本阶段不做的内容

为控制当前结构下的变更面，阶段 1 明确不做：

- Generic 通用模式
- 独立 FFmpeg 工作台
- Deno Hook
- 多语言与组件中心

这些能力保留在后续中长期路线中推进。

### 5A. Generic 通用模式第一版（中长期）

#### 目标

在保留 YouTube 专业模式的同时，为 yt-dlp 支持站点增加通用入口。

#### 原则

- Generic 模式与 YouTube 专业模式分开
- 不在第一版为通用站点做大量站点特化 UI
- 先做“尽量通用、尽量少假设”的模式

#### 实施步骤

1. 增加 `site_mode=youtube/generic`。
2. UI 上增加模式切换。
3. Generic 模式下允许任意受支持 URL。
4. 格式获取与元数据展示采用更保守字段集。
5. 若目标站点缺乏稳定字段，不阻塞下载主流程。
6. 记录 extractor 名称，方便后续问题诊断。

#### 验收

- 非 YouTube 的 yt-dlp 支持链接可以入队和下载
- 不破坏当前 YouTube 主流程

### 1F. 元数据与附件能力补齐

#### 目标

把 YTSage 中提到的封面、描述、info.json、章节相关能力从“部分具备”提升到“明确可选、可理解”。

#### 实施步骤

1. 单视频页与批量页都补出以下开关：
   - 写入封面文件
   - 嵌入封面
   - 写入描述
   - 写入 info.json
   - 嵌入章节
   - 导出章节文件
2. 对“嵌入”与“导出”做清晰区分。
3. 在摘要中显示附件与元数据输出项。

#### 验收

- 用户可明确知道自己会得到哪些附加文件
- 现有封面/元数据逻辑保持兼容

## 阶段 2：字幕系统专业化

### 目标

将当前单一字幕逻辑升级为完整字幕系统。

### 代码落点

   - `subtitle_format`

   - `--write-subs`
   - `--write-auto-subs`
   - `--sub-langs`
   - `--sub-format`

## 4A. 交付给开发人员的直接执行版

本节是写给实际开发人员的，不要求开发人员先理解完整产品演进史，只要求按顺序执行。

### 4A.1 开发人员必须先理解的事实

1. 当前程序不是通用下载器架构，而是 YouTube 专用架构。
2. 当前最安全的开发范围只有阶段 0 到阶段 4。
3. 阶段 5 及以后不是现在立刻开做的内容。
4. 当前最重要的目标不是“功能越多越好”，而是“现有 YouTube 主流程增强后仍稳定”。
5. 如果某个改动会同时影响 URL 识别、任务模型、历史仓库、调度器、页面结构，那么这个改动默认过大，必须拆小。

### 4A.2 开发人员阅读顺序

开始开发前，按这个顺序阅读：

1. 本文档
2. [`plans/YTSage功能吸收对照表.md`](./YTSage功能吸收对照表.md)
3. [`plans/能力覆盖清单_yt_dlp_ffmpeg.md`](./能力覆盖清单_yt_dlp_ffmpeg.md)
4. [`YCB.pyw`](../YCB.pyw)
5. [`core/youtube_models.py`](../core/youtube_models.py)
6. [`core/ytdlp_builder.py`](../core/ytdlp_builder.py)
7. [`core/download_manager.py`](../core/download_manager.py)
8. [`ui/pages/single_video.py`](../ui/pages/single_video.py)
9. [`ui/pages/batch_source.py`](../ui/pages/batch_source.py)
10. [`core/history_repo.py`](../core/history_repo.py)

### 4A.3 开发人员禁止事项

在阶段 0 到阶段 4 中，禁止直接做以下事情：

- 把项目重构成通用多站点下载器
- 把 FFmpeg 工作台和 YouTube 下载主流程一起改
- 把 Deno Hook 和下载调度器一起改
- 一次性重写 `YCB.pyw`
- 一次性重写 `ui/pages/single_video.py`
- 一次性改动任务模型、历史仓库和所有 UI 绑定
- 在没有回归验证前直接修改历史表结构并发布

### 4A.4 开发人员交付规则

每个阶段都必须产出以下交付物：

- 代码改动
- 对应阶段执行清单
- 对应阶段冒烟记录
- 已知问题列表

每个阶段内部建议拆成多个小批次提交，每个小批次只做一件事，例如：

- 只改 builder
- 只改 profile 字段
- 只改单视频页 UI
- 只改批量页 UI
- 只补日志与诊断

### 4A.5 开发人员验收规则

开发人员不能只说“代码写完了”，必须给出以下结果：

- 改了哪些文件
- 新增了哪些字段
- 新增了哪些命令参数
- 对现有功能有无兼容影响
- 手工验证了哪些链接或场景
- 还有哪些未验证风险

## 9. 风险与约束

### 9.1 参数漂移

yt-dlp 参数可能随版本变化，必须把参数映射尽量集中在 builder 层，避免散落在 UI 或 manager 中。

### 9.2 外部二进制依赖

yt-dlp、ffmpeg、deno 都是外部工具，必须有：

- 路径检测
- 版本检测
- 缺失提示
- 超时控制

### 9.3 高级参数风险

参数透传极易破坏主流程，必须做冲突检测，且日志中保留可复现信息。

### 9.4 Deno 与现有 Node 并存

短期内 Deno 适合作为插件与自动化层；现有 PO Token Node 脚本不建议在同一阶段强行替换，否则会扩大变量面。

### 9.5 UI 复杂度膨胀

新增功能优先放入“高级/展开区域”，避免单视频页和批量页一开始就变成密集参数面板。

### 9.6 当前结构适配风险

当前程序仍明显具备以下特征：

- Tkinter 单体应用壳
- YouTube 专用任务模型
- YouTube 专用历史仓库
- YouTube 专用 URL 检测与文案

因此本方案应理解为：

- 阶段 0 到阶段 4 是当前适合直接执行的路线
- 阶段 5 及以后是中长期演进路线
- 若要提前做 Generic 或 FFmpeg 工作台，必须先做中层抽象
