# 当前程序问题修复方案

## 1. 目标

本方案用于对当前 YouTube 下载器进行一轮**面向可用性、可维护性与可诊断性**的问题收口，供后续按步骤逐项修复调用。

本轮方案仅针对当前已识别出的真实问题，不扩散到全新大功能开发。修复目标集中在：
- 统一 GUI 交互认知
- 降低日志与状态提示误导
- 收口页面状态管理隐患
- 提升错误诊断可见性与可分类性
- 清理影响后续维护的冗余代码

---

## 2. 当前问题总表

### P1. 双日志区域导致用户认知混乱

位置：
- [`ui/download_tab.py`](ui/download_tab.py)
- [`ui/queue_tab.py`](ui/queue_tab.py)

现象：
- 单视频页 / 批量页右侧仍显示“实时日志”
- 但真正绑定到下载管理器的日志控件在 [`ui/queue_tab.py`](ui/queue_tab.py)
- 用户可能看到一个不更新或更新不一致的日志区域，误以为程序没有输出

影响：
- 日志入口不唯一
- 任务观察路径不清晰
- 降低可用性

优先级：高

### P2. 关闭窗口时的状态检查只依赖单一输入页引用

位置：
- [`JJH_download.pyw`](JJH_download.pyw)
- [`core/download_manager.py`](core/download_manager.py)
- [`ui/download_tab.py`](ui/download_tab.py)

现象：
- 当前只使用单个 `self.manager.input_frame`
- 该引用会被后创建的标签页覆盖
- `_on_close()` 只能检查一个页面实例的后台状态

影响：
- 关闭保护逻辑不稳定
- 后续页面异步能力增加后容易漏检
- 属于结构性隐患

优先级：高

### P3. 标题获取阶段日志文案与真实行为不完全一致

位置：
- [`core/download_manager.py`](core/download_manager.py)
- [`core/youtube_metadata.py`](core/youtube_metadata.py)

现象：
- [`fetch_title()`](core/youtube_metadata.py:384) 内部已经自行处理 cookies 重试策略
- 但下载管理器仍打印“尝试使用cookies重试...”之类文案
- 文案可能与实际执行路径不一致

影响：
- 误导用户判断问题来源
- 影响日志可信度

优先级：中高

### P4. 批量入队链路存在调试残留 / 冗余语句

位置：
- [`ui/pages/batch_source.py`](ui/pages/batch_source.py)

现象：
- 存在无意义语句：
  ```python
  task = self.manager.app.metadata_service and None
  ```

影响：
- 可读性差
- 后续维护者难以判断是否遗漏逻辑
- 影响代码整洁度

优先级：中

### P5. 未知错误占比仍偏高，自动诊断覆盖不足

位置：
- [`core/youtube_metadata.py`](core/youtube_metadata.py)
- [`core/download_manager.py`](core/download_manager.py)
- [`ui/history_actions.py`](ui/history_actions.py)

现象：
- 当前虽然已能透出部分原始日志
- 但很多 yt-dlp / YouTube 新报错仍会落入 unknown
- 用户仍需人工阅读原始日志判断问题类别

影响：
- 排障效率低
- 运行状态中心信息价值不足
- 失败样本难沉淀为稳定规则

优先级：中高

## 3. 修复原则

- 优先修复会直接影响用户操作认知的问题
- 优先修复低风险、高收益的问题
- 不在本轮引入额外复杂功能
- 保持当前“单视频页 / 批量页 + 独立下载队列页”的结构不被破坏
- 所有改动优先复用现有模块，不推翻阶段 F 已完成成果

---

## 4. 分步修复计划

### Step 1. 收口日志展示入口

目标：
- 让日志展示位置唯一
- 用户看到的日志与真实运行日志保持一致

修改建议：
- 精简 [`ui/download_tab.py`](ui/download_tab.py)
- 移除右侧“实时日志”区域
- 保留页面说明与“🧾 打开下载队列页”按钮
- 将日志唯一展示位置固定在 [`ui/queue_tab.py`](ui/queue_tab.py)

预期结果：
- 单视频页 / 批量页负责输入
- 队列页负责任务观察、日志和控制
- 用户心智模型统一

风险：低

验收标准：
- 单视频页和批量页不再显示独立日志框
- 所有下载日志只在“🧾 下载队列”页可见
- 任务执行时日志持续刷新正常

---

### Step 2. 修正标题获取阶段日志文案

目标：
- 让日志文案准确反映真实行为
- 避免“已重试 / 未重试 / 是否使用 cookies”表达混乱

修改建议：
- 调整 [`core/download_manager.py`](core/download_manager.py) 中标题获取阶段的日志语句
- 明确区分：
  - 标题获取失败
  - 已返回诊断摘要
  - 将继续进入下载阶段
  - 是否实际使用 cookies 成功获取标题

预期结果：
- 用户从日志即可理解程序当前处于什么阶段
- 减少误判为 cookies 问题或网络问题

风险：低

验收标准：
- 日志内容与 [`fetch_title()`](core/youtube_metadata.py:384) 的真实行为一致
- 不再出现误导性“尝试重试”提示

---

### Step 3. 清理批量入队冗余代码

目标：
- 去掉无意义残留
- 提高批量页代码可读性

修改建议：
- 删除 [`ui/pages/batch_source.py`](ui/pages/batch_source.py) 中无意义语句
- 顺带检查同函数中是否还有类似重构残留

预期结果：
- 批量入队链路更清晰
- 降低后续误改风险

风险：极低

验收标准：
- 批量入队功能行为不变
- 代码中不再有无意义占位语句

---

### Step 4. 重构页面状态登记与关闭保护逻辑

目标：
- 让关闭窗口时能检查所有相关页面的异步状态
- 消除 `input_frame` 单点覆盖隐患

修改建议：
- 不再只依赖单个 `self.manager.input_frame`
- 在 [`JJH_download.pyw`](JJH_download.pyw) 中统一登记页面实例，例如：
  - 单视频页实例
  - 批量页实例
- `_on_close()` 中按页面逐个检查是否存在：
  - 批量解析中
  - 批量入队处理中
  - 后续新增的异步动作

可选实现方向：
- 方向 A：应用层维护 `self.page_refs`
- 方向 B：管理器维护 `self.registered_frames`

推荐：方向 A
原因：
- 页面生命周期本来就在应用层创建
- 关闭逻辑也在 [`JJH_download.pyw`](JJH_download.pyw)
- 职责更清晰

预期结果：
- 关闭保护逻辑不再依赖页面创建顺序
- 后续扩展异步页面时更稳

风险：中

验收标准：
- 页面引用管理清晰
- 关闭窗口时能正确识别所有活动后台状态
- 不引入新的页面初始化错误

---

### Step 5. 扩充错误诊断规则

目标：
- 降低 unknown 比例
- 提升认证 / 网络 / 环境 / 提取失败等问题的自动分类能力

修改建议：
- 扩展 [`detect_auth_diagnostic()`](core/youtube_metadata.py:34) 规则集
- 优先补充以下类别：
  - YouTube 限流 / bot check
  - cookies 失效 / 登录要求
  - 403 / forbidden
  - nsig / JS challenge / 提取器失效
  - 代理 / DNS / 超时 / 连接中断
  - ffmpeg / 外部依赖缺失
  - 视频不可用 / 地区限制 / 私有 / 会员专属

同时建议：
- 为标题获取失败增加更明确的阶段字段
- 让 [`ui/history_actions.py`](ui/history_actions.py) 的运行状态中心能显示更准确的问题类型

预期结果：
- 更多错误能自动归类
- 用户无需完全依赖人工阅读原始日志

风险：中

验收标准：
- 常见错误样本能被更稳定识别
- unknown 比例下降
- 运行状态与日志提示更一致

---

## 5. 推荐执行顺序

建议后续按以下顺序调用修复：

1. Step 1：收口日志展示入口
2. Step 2：修正标题获取阶段日志文案
3. Step 3：清理批量入队冗余代码
4. Step 4：重构页面状态登记与关闭保护逻辑
5. Step 5：扩充错误诊断规则

这样安排的原因：
- 前三步改动小、收益高、风险低，适合先快速收口
- 第四步属于结构修复，需要在 UI 稳定后做
- 第五步属于诊断增强，适合放在结构稳定后逐步补强
