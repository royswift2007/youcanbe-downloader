# 项目代码审查清单

本清单用于对当前项目进行系统性的代码审查、优化和修复跟踪。

使用方式：
- 审查时按 `P0 -> P1 -> P2` 的顺序执行。
- 每完成一项检查，将对应条目标记为已完成。
- 如果发现问题，在“审查记录”部分追加问题描述、风险等级、影响范围和修复建议。
- 如果某项不适用于当前模块，直接标记为不适用并写明原因。

## 一、审查优先级

### P0：必须优先检查

#### 1. 下载主流程
- [ ] 下载任务是否可能出现假成功、卡死、重复执行、无响应。
- [ ] 下载失败时是否会留下错误状态但界面显示成功。
- [ ] 下载完成回调、历史写入、Hook 触发是否可能重复执行。
- [ ] 异常退出后是否存在残留任务状态。
- [ ] 标题获取失败当前会写失败历史，但任务仍继续进入下载阶段；需检查这是否会造成“一次任务产生两条失败/成功混合记录”。

#### 2. 任务状态一致性
- [ ] 创建、排队、启动、暂停、继续、取消、失败、完成状态是否定义清晰。
- [ ] 状态迁移是否存在非法跳转。
- [ ] 关闭窗口时会将运行中和等待中任务强制重置为 `WAITING` 后保存快照，需检查是否会覆盖真实状态并导致误恢复。
- [ ] 删除任务时是否同步清理内部状态和界面状态。
- [ ] 取消中的任务是否可能继续写入进度、日志或历史记录。
- [ ] `run_task()` 结束后会将成功/失败/停止任务重新放回 `task_queue`；需确认该设计不会导致已结束任务与等待任务混杂产生二次处理风险。

#### 3. 历史记录与配置数据安全
- [ ] 历史记录写入是否可能损坏 JSON 文件。
- [ ] 并发写入时是否存在冲突。
- [ ] JSON 历史、SQLite 历史、待恢复快照、窗口状态文件都直接覆盖写入，需检查是否需要原子写入和临时文件替换。
- [ ] 空文件、损坏文件、缺字段文件是否能恢复。
- [ ] 配置文件升级兼容是否完善。
- [ ] 历史仓库在 DB 不可用时会自动回退 JSON，需要检查回退后读取条数、字段完整性和排序是否一致。

#### 4. 外部组件与命令安全
- [ ] `yt-dlp` 参数构造是否安全。
- [ ] `ffmpeg` 参数构造是否安全。
- [ ] 外部组件路径是否经过校验。
- [ ] 是否存在命令注入、参数污染、未过滤输入直接拼接命令的问题。
- [ ] 日志中的命令摘要、代理地址、高级参数、cookies 模式、浏览器来源是否暴露过多敏感上下文。

#### 5. 文件系统安全
- [ ] 输出目录是否合法。
- [ ] 文件名是否经过清洗。
- [ ] `custom_filename` 作为输出模板的一部分时，是否可能包含非法字符、保留名、相对路径片段或空白名。
- [ ] 是否可能误覆盖已有文件。
- [ ] 删除任务关联文件时，基于标题/自定义文件名模糊匹配删除，需检查是否可能误删同目录其他文件。
- [ ] 是否存在路径穿越风险。
- [ ] 临时文件、`.ytdl` 文件、区段回退产生的 `ycb_section_full_*` 文件是否能被正确清理。

#### 6. GUI 稳定性
- [ ] 长时间任务是否阻塞主线程。
- [ ] 下载过程中界面是否可能假死。
- [ ] 异常是否能反馈到界面层。
- [ ] 后台线程更新 UI 是否安全。
- [ ] 语言切换会销毁并重建 notebook / top_bar / bottom_bar，需检查重建期间日志刷新、任务列表刷新是否会访问失效控件。

#### 7. 异常场景处理
- [ ] 网络中断时是否有明确处理。
- [ ] 磁盘满时是否有明确处理。
- [ ] 路径无权限时是否有明确处理。
- [ ] 外部组件缺失时是否有明确处理。
- [ ] 下载无输出/无进度超时被 watchdog 终止时，是否能给出明确原因并正确收尾。
- [ ] 程序异常是否会导致未关闭句柄、未终止进程或脏状态。

#### 8. 打包与安装可用性
- [ ] 打包结果是否包含运行所需全部依赖。
- [ ] 安装器首次安装后是否可以正常启动。
- [ ] 升级安装是否可能破坏旧版本配置或数据。
- [ ] 卸载时是否会误删用户数据。

#### 9. 敏感数据保护
- [ ] 日志是否包含 cookie、token、认证头等敏感信息。
- [ ] 日志是否记录完整命令行，导致认证数据泄漏。
- [ ] 本地持久化数据是否存储了不必要的敏感字段。
- [ ] PO Token 的 `visitor_data` / `po_token`、代理配置、cookies 浏览器来源是否可能通过日志、诊断导出或异常信息泄漏。

### P1：高优先级优化项

#### 1. 结构与职责划分
- [ ] 模块职责是否清晰。
- [ ] 是否存在超长函数、超大类、强耦合逻辑。
- [ ] UI 层、业务层、外部调用层是否职责混杂。

#### 2. 性能与资源占用
- [ ] 是否存在重复读取配置、历史记录、元数据的问题。
- [ ] 是否存在不必要的重复构造参数。
- [ ] 是否存在内存增长、对象残留、线程泄漏。
- [ ] 剪贴板轮询、日志轮询、状态刷新 `after()` 调度是否会在长时间运行后累积负担。
- [ ] 大量历史记录下性能是否退化明显。

#### 3. 并发与回调稳定性
- [ ] 回调是否可能重复触发。
- [ ] 并发下载是否可能导致共享状态竞争。
- [ ] 队列调度是否稳定。
- [ ] 暂停、恢复、取消时是否存在竞态条件。

#### 4. 错误处理与日志质量
- [ ] 是否区分用户输入错误、环境错误、程序错误。
- [ ] 日志级别是否合理。
- [ ] 日志是否包含足够上下文用于排查。
- [ ] 是否存在吞异常或模糊错误提示。
- [ ] `record_runtime_issue`、顶部状态条、日志窗口三套错误出口的语义是否统一。

#### 5. 后处理链路
- [ ] 音视频合并顺序是否正确。
- [ ] 元数据写入时机是否正确。
- [ ] 封面、转码、重命名之间是否存在顺序冲突。
- [ ] 某一步失败后是否会污染整体结果。

#### 6. 设置项与输入校验
- [ ] 设置项默认值是否合理。
- [ ] 非法输入是否会被拦截。
- [ ] `download_retry_var`、`download_concurrent_var`、`download_speed_limit_var` 当前通过 `int()` 直接转换，需检查非法输入是否会在同步设置时抛异常。
- [ ] 高级参数是否可能破坏默认行为。
- [ ] 设置项保存后是否立刻生效且不引入副作用。

### P2：维护性与一致性

#### 1. 代码质量
- [ ] 命名是否一致。
- [ ] 是否存在魔法值、硬编码路径、硬编码平台逻辑。
- [ ] 是否有可以抽取的公共逻辑。
- [ ] 复杂逻辑处是否需要补充注释。

#### 2. UI 与文案一致性
- [ ] 中文和英文文案是否一致。
- [ ] 提示信息是否准确。
- [ ] 按钮状态和功能含义是否一致。
- [ ] 错误提示是否对用户可理解。

#### 3. 文档一致性
- [ ] `README.md` 是否与当前实现一致。
- [ ] `README_EN.md` 是否与当前实现一致。
- [ ] 使用说明是否覆盖当前功能。
- [ ] 安装与排障说明是否仍然有效。

## 二、项目定制审查点

### 1. 应用启动与运行目录
- [ ] 区分开发态与 PyInstaller 打包态时，`base_path`、`APP_DATA_DIR`、`_MEIPASS` 资源寻址是否始终正确。
- [ ] `LOCALAPPDATA` / `APPDATA` 不可写时，回退到 `base_path` 是否会引起权限问题。
- [ ] `get_resource_path()` 在资源缺失时返回回退路径，是否会导致“路径存在但资源实际未打包”的隐性错误。
- [ ] `install_prefs.json` 保存在安装目录，而运行数据保存在 `APP_DATA_DIR`，两套目录策略是否一致且符合预期。

### 2. 任务恢复与持久化快照
- [ ] `pending_tasks.json` 在保存时是否具备原子性，程序异常退出时是否可能写出半截 JSON。
- [ ] 启动恢复未完成任务时，重复任务 ID、缺失字段、损坏快照是否都能安全跳过。
- [ ] 恢复后的任务被统一重置为 `WAITING`、`0%`、`0 M/s`，是否会掩盖真实中断位置并影响用户判断。
- [ ] 恢复任务时如果 `save_path` 为空，会回退到当前共享目录；需要检查这是否会改变原任务输出位置。
- [ ] 恢复归档任务时 `archive_root` / `archive_subdir` 与 `resolve_output_dir()` 的优先级是否稳定。

### 3. 历史记录双存储策略
- [ ] 项目同时使用 JSON 历史和 SQLite 历史，需确认两套数据是否存在写入顺序不一致、字段不一致、回退逻辑不一致的问题。
- [ ] SQLite 初始化失败后会降级，需要确认降级路径是否仍然完整保留历史功能。
- [ ] SQLite schema 通过运行时 `ALTER TABLE` 升级，需检查旧版本数据库升级失败时是否有回滚或提示。
- [ ] 历史记录中 `status`、`failure_stage`、`failure_summary`、`failure_detail` 的语义是否统一，避免 UI 侧展示不一致。
- [ ] `HISTORY_JSON_LIMIT` 与数据库长期存储策略是否一致，避免“列表展示”和“底层存储”行为不同。

### 4. 组件检测与版本门槛
- [ ] `yt-dlp`、`ffmpeg`、`deno` 的版本检测完全依赖命令输出格式，需要检查输出格式变化时是否会出现误判。
- [ ] `ffmpeg -version` 只校验首行前缀，需确认是否足以识别异常二进制或损坏可执行文件。
- [ ] `deno` 路径会回退到 `PATH` 或固定字符串 `deno`，需要检查误报“已配置但不可执行”的情况。
- [ ] 组件检测超时固定为 6 秒，需确认在慢机器或安全软件拦截场景下不会频繁误判。
- [ ] 诊断导出 JSON 是否可能泄露本地绝对路径或用户环境信息。

### 5. yt-dlp 参数构造的项目特性
- [ ] `cookies_mode=file/browser` 与 `task.needs_cookies` 的组合逻辑是否覆盖全部场景，避免明明需要 cookies 却未注入。
- [ ] `custom_filename` 直接参与输出模板拼接，需要检查非法文件名、保留字符、空白名和目录穿透问题。
- [ ] 音频下载使用 `-x --audio-format`，视频下载使用 `--merge-output-format`；需检查两套分支下缩略图、字幕、元数据选项是否相互冲突。
- [ ] `merge_output_format=webm` 与 `h264_compat=True` 已显式互斥，需要继续检查其他互斥组合是否也已拦截。
- [ ] `advanced_args` 通过策略解析后注入，需检查是否仍可能覆盖关键默认参数。
- [ ] `use_po_token` 开启后会拼接 `--extractor-args`，需要检查它与 cookies、代理、高级参数之间的兼容性。
- [ ] `download_sections`、`sponsorblock`、字幕模式、封面嵌入、章节嵌入的组合是否会触发 yt-dlp/ffmpeg 后处理异常。

### 6. PO Token 工具链
- [ ] PO Token 依赖 Node.js、本地脚本目录和后台修复线程，需检查工具缺失时是否真正做到静默降级。
- [ ] `_TOOLS_DIR`、`_SCRIPT_PATH` 在打包态下是否一定可用，未打包到发行版时状态提示是否准确。
- [ ] `get_token()` 的缓存、过期、修复、退避重试逻辑是否会造成多线程争用或重复修复。
- [ ] `repair()` 与自动修复同时触发时，状态机是否可能出现错乱。
- [ ] Token 生成失败后是否会污染后续普通下载流程，或在日志中泄露 visitor_data / po_token。

### 7. GUI 与线程调度模型
- [ ] Tk 主线程通过 `root.after()` 接收后台线程更新，需要检查所有 UI 更新是否都通过安全调度进入主线程。
- [ ] `process_log_queue()` / `update_list()` 这类轮询刷新是否会在长时间运行后造成 UI 卡顿或无上限调度。
- [ ] 静音消息框 `SilentMessagebox` 使用 `grab_set()` 和 `wait_window()`，需检查模态框嵌套时是否可能锁死交互。
- [ ] `startupinfo`、隐藏控制台窗口、子进程启动参数在 Windows 非 Windows 下是否行为一致。

### 8. 媒体工具链与本地 ffmpeg 任务
- [ ] `MediaJobManager` 与下载任务管理器并行存在，需要检查它们的日志、状态、并发策略是否一致。
- [ ] `build_ffmpeg_command()` 会拼接 `vf_custom` / `af_custom` / `extra_args`，需检查策略校验是否足以避免危险参数覆盖。
- [ ] 字幕烧录时 `_build_subtitle_filter()` 只转义部分字符，需要检查包含引号、冒号、反斜杠、Unicode 路径时的兼容性。
- [ ] `remux` 分支先设置 `-c copy` 再附加编码参数，需检查参数冲突时 ffmpeg 最终行为是否符合预期。
- [ ] 媒体任务失败、中止、输出文件已存在时，是否会留下半成品或覆盖原文件。

### 9. 配置与窗口状态持久化
- [ ] `window_pos.json` 保存未使用原子写入，需检查异常退出时配置损坏后的恢复能力。
- [ ] `is_geometry_visible()` 只基于单屏尺寸判断，需检查多显示器切换、缩放变化后的窗口恢复体验。
- [ ] `ui_state` 与窗口位置绑定存储，需检查未来字段扩展时是否容易互相污染。

## 三、按模块审查

### `YCB.pyw`
- [ ] UI 初始化是否过重。
- [ ] 是否混入大量下载业务逻辑、依赖检测、任务恢复、状态聚合和安装器语言消费逻辑。
- [ ] UI 状态刷新是否覆盖全部关键状态。
- [ ] 关闭流程 `_on_close()` 中的状态重写、快照保存、停止任务和延迟销毁顺序是否安全。
- [ ] 异常是否可以传递到用户可见层。

### `core/download_manager.py`
- [ ] 任务生命周期定义是否完整。
- [ ] 状态迁移逻辑是否集中且可验证。
- [ ] 标题获取、下载执行、区段 fallback、成功/失败历史写入、Hook 触发之间是否存在重复收尾逻辑。
- [ ] Windows 专用 `taskkill` 停止逻辑在非 Windows 下是否存在兼容性问题。
- [ ] 删除任务时的关联文件匹配策略是否足够精确。
- [ ] 异常退出时资源释放是否完整。

### `core/history_repo.py`
- [ ] 读写是否原子化。
- [ ] 损坏数据恢复逻辑是否合理。
- [ ] JSON 旧字段迁移、中文键兼容、布尔值兼容是否覆盖完整。
- [ ] 去重、排序、更新逻辑是否一致。
- [ ] `save_task()` / `save_failed_task()` 同时写 DB 和 JSON，需确认任一失败时不会造成用户感知错乱。
- [ ] 高并发或频繁写入下是否稳定。

### `core/components_manager.py`
- [ ] 外部组件检测是否可靠。
- [ ] 组件检测失败信息是否足够区分“未配置”“文件不存在”“版本过旧”“输出异常”。
- [ ] 组件下载失败后的回滚逻辑是否完整。
- [ ] 组件版本判断是否合理。
- [ ] 导出的诊断文件是否应该脱敏本地路径。
- [ ] 路径切换后是否会遗留旧状态。

### `core/ffmpeg_builder.py`
- [ ] 参数构造是否严格。
- [ ] 文件路径和特殊字符处理是否安全。
- [ ] `vf_custom` / `af_custom` 与内置滤镜链拼接时索引是否正确，当前存在使用 `vf_custom[1]` / `af_custom[1]` 的实现，需重点检查是否误取参数名而不是滤镜值。
- [ ] `remux` / `trim` / `concat` 先设置 `-c copy` 又追加编码参数时，需确认最终行为是否符合预期。
- [ ] 不同媒体类型分支是否覆盖完整。
- [ ] 转码、合并、封面、元数据逻辑是否冲突。

### `core/ytdlp_builder.py`
- [ ] 参数拼装是否和配置一致。
- [ ] `cookies_mode=browser` 时当前无论任务是否需要都会注入浏览器 cookies，需确认这是否符合设计。
- [ ] cookies、headers、认证参数是否按需注入。
- [ ] `write_subs` 变量已读取但未参与最终条件，需检查字幕写入逻辑是否与 UI 配置一致。
- [ ] 高级参数是否影响默认参数正确性。
- [ ] 参数互斥关系是否明确。

### `core/youtube_metadata.py`
- [ ] 元数据字段缺失时是否可容错。
- [ ] 字段映射是否依赖外部结构过强。
- [ ] 标题、作者、时长、封面等字段清洗是否统一。
- [ ] 外部接口结构变更时是否容易失效。

### `core/media_jobs.py`
- [ ] 后处理任务编排是否清晰。
- [ ] 单步失败是否正确中断或降级。
- [ ] 任务结束后当前会重新放回 `job_queue`，需确认已完成/失败任务不会影响后续调度和删除逻辑。
- [ ] 停止任务同样使用 Windows 专用 `taskkill`，需检查跨平台兼容性。
- [ ] 状态同步是否正确。
- [ ] 是否可能留下半处理文件。

### `core/settings.py`
- [ ] 默认配置是否合理。
- [ ] 配置缺失时是否可恢复。
- [ ] 多显示器、分辨率变化、缩放变化后窗口位置恢复是否仍可见。
- [ ] 配置升级兼容是否明确。
- [ ] 持久化时是否安全。

### `installer/setup.iss`
