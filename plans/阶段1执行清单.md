# 阶段1执行清单

## 1. 阶段定位

阶段 1 是当前项目最重要的“核心增强阶段”。

它的任务不是做大改造，而是把最影响下载成功率、可控性和专业度的能力补齐，同时保持当前 YouTube 主流程稳定。

本阶段只做 5 件事：

- 区段下载
- SponsorBlock
- 代理支持与网络诊断
- 高级参数透传
- 元数据与附件能力补齐

---

## 2. 本阶段完成标准

只有满足以下全部条件，阶段 1 才算完成：

- 支持 `--download-sections`
- 支持 SponsorBlock
- 支持代理参数 `--proxy`
- 支持高级参数透传并做冲突检测
- 元数据与附件能力在 UI 中明确可控
- 单视频页与批量页字段保持一致
- 至少完成一轮阶段 1 冒烟

---

## 3. 本阶段不做什么

本阶段明确不做：

- Generic 模式
- 通用站点入口
- 独立 FFmpeg 本地媒体工作台
- Deno Hook
- 大规模设置中心重构
- 历史仓库通用化

如果开发人员在阶段 1 中开始做这些内容，视为超范围开发。

---

## 4. 前置条件

进入阶段 1 前，必须满足：

- 阶段 0 已完成
- 最小测试骨架已存在
- 当前主流程下载仍然正常
- builder 的最小测试已经可运行

## 8. 本阶段新增字段建议

本阶段只允许给 `YouTubeDownloadProfile` 增加与当前 YouTube 下载主流程直接相关的字段。

建议新增：

- `download_sections`
- `sponsorblock_enabled`
- `sponsorblock_categories`
- `proxy_url`
- `advanced_args`
- `cookies_mode`
- `cookies_browser`
- 如有必要，再补充元数据/附件导出字段的缺口

本阶段不要加入：

- Generic 模式字段
- 本地媒体处理字段
- Hook 配置字段
- 复杂 headers / referer / user-agent 体系

---

## 9. 建议实施顺序

不要并行乱改，严格按顺序推进：

1. 先改 `YouTubeDownloadProfile`
2. 再改 builder
3. 再改输入校验
4. 再改单视频页
5. 再改批量页
6. 再补测试
7. 最后做冒烟

## 10. 子任务 A：区段下载

### 目标

支持 yt-dlp 的 `--download-sections`，为用户提供基础区段下载能力。

### 必改文件

- [`core/youtube_models.py`](../core/youtube_models.py)
- [`core/ytdlp_builder.py`](../core/ytdlp_builder.py)
- [`ui/input_validators.py`](../ui/input_validators.py)
- [`ui/pages/single_video.py`](../ui/pages/single_video.py)
- [`ui/pages/batch_source.py`](../ui/pages/batch_source.py)

### 详细步骤

1. 在 profile 中新增区段字段。
   建议第一轮使用简单文本格式。

2. 在输入校验层增加时间格式校验。
   建议先支持：
   - `HH:MM:SS`
   - `MM:SS`

3. 在 builder 中注入 `--download-sections`。

4. 单视频页增加：
   - 开始时间
   - 结束时间
   - 可选说明文本

5. 批量页第一轮不做每条目不同区段，只做统一设置。

6. 在日志摘要中显示区段设置。

### 验收标准

- 非法时间输入不能入队
- 命令中能看到 `--download-sections`
- 区段下载结果大致符合预期

### 本子任务不要做

- 不做复杂多段编辑器
- 不做自动 FFmpeg fallback
- 不做波形预览

---

## 11. 子任务 B：SponsorBlock

### 目标

支持 SponsorBlock 去片段能力，但保持默认保守。

### 必改文件

- [`core/youtube_models.py`](../core/youtube_models.py)
- [`core/ytdlp_builder.py`](../core/ytdlp_builder.py)
- [`ui/pages/single_video.py`](../ui/pages/single_video.py)
- [`ui/pages/batch_source.py`](../ui/pages/batch_source.py)

### 详细步骤

1. 增加是否启用 SponsorBlock 的字段。
2. 增加类别字段。
3. builder 中根据字段注入 SponsorBlock 参数。
4. UI 上增加：
   - 启用开关
   - 类别输入或下拉
5. 默认关闭。
6. 摘要中显示已启用 SponsorBlock。

### 默认建议

- 第一轮默认不要勾选 `all`
- 第一轮建议允许开发者先用文本类别输入，后续再做更友好的类别选择器

### 验收标准

- 开启后命令中存在 sponsorblock 参数
- 默认关闭时行为与当前版本一致

### 本子任务不要做

- 不做复杂分类可视选择器
- 不做片段预览

---

## 12. 子任务 C：代理支持与网络诊断

### 目标

让代理成为显式能力，并把常见网络失败变成可操作提示。

### 必改文件

- [`core/youtube_models.py`](../core/youtube_models.py)
- [`core/ytdlp_builder.py`](../core/ytdlp_builder.py)
- [`core/download_manager.py`](../core/download_manager.py)
- [`core/youtube_metadata.py`](../core/youtube_metadata.py)
- [`ui/pages/single_video.py`](../ui/pages/single_video.py)
- [`ui/pages/batch_source.py`](../ui/pages/batch_source.py)

### 详细步骤

1. 新增代理字段。
2. builder 注入 `--proxy`。
3. UI 增加代理输入框。
4. 在元数据获取与下载失败链路中识别典型网络问题。

建议至少识别：

- 代理连接失败
- DNS 失败
- 超时
- 403
- 认证失败

5. 日志中给出建议，不只输出原始错误。

### 验收标准

- 代理地址可进入命令
- 代理相关失败能给出提示

### 本子任务不要做

- 不做系统级代理管理
- 不做复杂代理池

---

## 13. 子任务 D：高级参数透传

### 目标

用“可控透传”补齐剩余 yt-dlp 官方能力，而不是继续堆 UI 控件。

### 必改文件

- [`core/youtube_models.py`](../core/youtube_models.py)
- [`core/ytdlp_builder.py`](../core/ytdlp_builder.py)
- [`ui/pages/single_video.py`](../ui/pages/single_video.py)
- [`ui/pages/batch_source.py`](../ui/pages/batch_source.py)
- [`ui/input_validators.py`](../ui/input_validators.py)

### 详细步骤

1. profile 中增加附加参数字段。
2. UI 增加高级参数文本框。
3. 实现参数解析器。
4. 建立冲突参数黑名单。
5. builder 在最后阶段追加合法参数。
6. 日志中输出“用户附加参数”。

### 最低黑名单

- `-o`
- `--output`
- `--ffmpeg-location`
- `--cookies`
- `--cookies-from-browser`
- `--exec`
- `--config-locations`

### 验收标准

- 合法附加参数进入命令
- 冲突参数会被拦截
- 拦截时有明确提示原因

### 本子任务不要做

- 不做无限制透传
- 不允许覆盖关键路径参数

---

## 14. 子任务 E：元数据与附件能力补齐

### 目标

把当前已有但暴露不完整的附件能力补成一致、明确、可理解的能力。

### 必改文件

- [`core/youtube_models.py`](../core/youtube_models.py)
- [`core/ytdlp_builder.py`](../core/ytdlp_builder.py)
- [`ui/pages/single_video.py`](../ui/pages/single_video.py)
- [`ui/pages/batch_source.py`](../ui/pages/batch_source.py)

### 详细步骤

1. 对齐单视频页与批量页的元数据/附件开关。
2. 补齐以下能力的 UI：
   - 写缩略图文件
   - 嵌入缩略图
   - 写描述
   - 写 info.json
   - 嵌入章节
   - 章节导出
3. 摘要中展示启用项。
4. builder 参数与 UI 名称保持一致。

### 验收标准

- 用户能看懂会输出哪些附加文件
- 单视频页与批量页能力一致

## 16. 阶段1冒烟清单

至少跑以下场景：

1. 单视频正常下载
2. 区段下载
3. 启用 SponsorBlock 下载
4. 代理下载
5. 音频下载 + 附件输出
6. 批量页统一设置入队
7. 高级参数透传下载
8. 冲突参数被拦截

## 17. 阶段验收清单

验收人必须检查：

- 是否只做了阶段 1 范围内的能力
- profile 字段是否过度膨胀
- builder 是否能稳定拼装命令
- 单视频页和批量页字段是否一致
- 旧下载能力是否被破坏
- 冒烟是否真实执行

---

- 区段下载可用
- SponsorBlock 可用
- 代理可用
- 高级参数透传可用
- 元数据与附件能力补齐
- 阶段 1 冒烟通过
