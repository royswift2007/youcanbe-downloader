# 阶段 F 执行清单

## 1. 阶段定位

阶段 F 面向 **归档系统与运维能力增强**，目标是在 [`plans/阶段E结项与冒烟记录.md`](plans/阶段E结项与冒烟记录.md) 已完成的认证中心、历史数据库与稳定性底座之上，继续补齐 YouTube 下载器的归档落盘、结果可见、运维可诊断与链路稳定性能力，使产品从“可用下载器”进一步升级为“更易维护的 YouTube 归档工具”。

本阶段优先完成：
- 归档目录与命名规则落地
- 同步 / 批量处理结果展示与错误摘要
- 基础运维入口与关键状态提示
- 下载与批量链路稳定性增强
- 阶段 F 冒烟验证与结项记录

本阶段明确删除：
- 频道 / 播放列表订阅源管理
- 订阅同步服务
- 订阅中心 GUI
- 与订阅相关的数据库结构与运维入口

本阶段暂不提前展开：
- 云端同步
- 多账户隔离
- 分布式任务调度
- 复杂规则脚本引擎
- 全自动后台常驻服务

---

## 2. 执行原则

- 必须复用阶段 D 的批量来源模型 [`core/youtube_models.py`](core/youtube_models.py) 与阶段 E 的历史数据库 [`core/history_repo.py`](core/history_repo.py)
- 必须保持单视频页 [`ui/pages/single_video.py`](ui/pages/single_video.py) 与批量页 [`ui/pages/batch_source.py`](ui/pages/batch_source.py) 主流程不被误伤
- 增量判断优先基于 `video_id`，其次回退到标准化 URL，不依赖标题字符串
- 归档规则优先实现“可预测、可解释、可复现”，避免过早做高度自由配置
- GUI 相关增强优先放在现有页面或已有入口内完成，不再新增订阅中心
- 阶段 F 完成前，必须进行“批量处理、重复跳过、归档落盘、失败可见”的完整冒烟验证

---

## 3. 执行清单

### F1. 归档目录与命名规则

- [ ] 为批量任务与下载结果设计最小归档规则，至少支持按频道、来源类型、日期三级组织输出目录
- [ ] 在现有任务入队或下载执行链路中增加归档目标路径生成逻辑，避免全部堆积在单一保存目录
- [ ] 在历史数据库中记录归档输出路径与归档上下文，便于后续检索与复盘
- [ ] 对路径非法字符、超长路径、缺失标题等场景提供回退策略
- [ ] 确保阶段 C / D 的手动下载链路不被新的归档规则强制改变

### F2. 批量处理结果与错误摘要可视化

- [ ] 在现有 GUI 中提供最小结果查看入口，展示新增、跳过、失败摘要
- [ ] 为最近一次批量处理错误提供可视化摘要，避免用户只能查看日志
- [ ] 对认证失败、网络失败、源地址失效、数据库失败、入队失败等情况提供清晰提示
- [ ] 统一批量处理日志风格，复用阶段 E 的错误分类与认证提示能力
- [ ] 确保结果展示不会干扰现有单视频和批量下载主流程

### F3. 基础运维入口增强

- [ ] 在现有页面或菜单中补充最小运维动作入口，如重新解析、清理异常状态、查看最近错误摘要
- [ ] 为历史数据库不可用、cookies 认证异常、关键依赖缺失等情况提供明确可见提示
- [ ] 将关键运维信息尽量沉淀到已有入口，避免新建独立复杂页面
- [ ] 统一运维提示文案与错误等级，降低用户理解成本
- [ ] 保证运维入口的行为可回退、可解释、可记录

### F4. 下载与批量链路稳定性增强

- [ ] 防止批量处理线程与下载线程在 UI 更新上互相干扰
- [ ] 对重复点击、并发触发、关闭窗口时仍有处理任务等场景给出最小保护
- [ ] 为关键处理步骤保留最近错误摘要，便于 GUI 与日志复用
- [ ] 确保阶段 E 已验证的认证中心、历史库、去重逻辑继续可用
- [ ] 对真实使用中的异常链路进行一次针对性回归修补

### F5. 阶段 F 冒烟验证

- [ ] 验证程序启动正常，主界面可正常进入单视频页与批量页
- [ ] 验证批量解析与入队流程可正常产生新增 / 跳过结果
- [ ] 验证历史中已存在的视频会被正确跳过，不重复入队
- [ ] 验证一个失败场景能输出明确摘要并在 GUI 可见
- [ ] 验证归档路径生成逻辑在常见标题 / 频道名下可稳定工作
- [ ] 验证阶段 C / D / E 已完成的主流程未被阶段 F 改动破坏

### F6. 阶段 F 文档收口

- [ ] 记录归档路径规则、命名规则与回退策略
- [ ] 记录批量结果摘要与错误分类展示方式
- [ ] 记录运维入口位置与典型使用说明
- [ ] 记录真实样本与失败样本的验证结果
- [ ] 编写正式的 [`plans/阶段F结项与冒烟记录.md`](plans/阶段F结项与冒烟记录.md)
- [ ] 明确哪些能力留待后续阶段，如定时后台任务、规则脚本化、全文检索历史中心

## 4. 推荐实施顺序

1. 落地最小归档路径与命名规则
2. 补充批量处理结果展示与错误摘要
3. 增强现有入口中的基础运维能力
4. 补强下载与批量链路稳定性
5. 执行阶段 F 冒烟验证与修复
6. 编写阶段 F 结项文档

---

## 5. 阶段 F 完成标准

满足以下条件后，阶段 F 才视为完成：
- 已具备最小归档目录与命名策略
- 已能在 GUI 中看到批量处理结果、错误摘要或关键异常提示
- 已具备最小运维入口，能帮助定位常见异常
- 下载与批量链路的关键稳定性问题已完成一轮收口
- 已完成阶段 F 冒烟验证并形成正式记录文档
- 文档中不再保留订阅中心、订阅同步、订阅仓储等已撤销方案
