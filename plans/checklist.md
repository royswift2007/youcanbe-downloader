# yt-dlp + FFmpeg + Deno + Python 媒体下载器 — 检查与验收清单

> 适用场景：以 YouTube 为核心的下载器，基于 yt-dlp + FFmpeg + Deno，使用 Python 实现。

---

## 一、环境依赖检查

在任何功能测试之前，先验证所有工具链是否正确安装并可被 Python 调用：

- **yt-dlp**：运行 `yt-dlp --version`，确认版本号输出正常
- **FFmpeg / FFprobe**：运行 `ffmpeg -version` 和 `ffprobe -version`，确认两个二进制都存在
  > ⚠️ 注意：pip 安装的 `ffmpeg-python` 包不等同于实际的 FFmpeg 可执行文件
- **Deno**：运行 `deno --version`，确认 Deno runtime 可用
- **Python 依赖**：运行 `pip list` 确认 `yt-dlp` 等库版本符合要求
- **`ffmpeg_location` 路径配置**：在 Python 的 `ydl_opts` 中确认 `ffmpeg_location` 指向正确的二进制目录

---

## 二、核心下载功能验收

针对最基础的下载路径进行逐项验收：

- **单视频下载**：下载一个普通 YouTube 视频，检查文件是否完整生成
- **播放列表下载**：使用 `noplaylist: False` 模式下载一个小型播放列表，验证所有子项均被处理
- **格式选择**：测试 `bestvideo+bestaudio/best` 格式合并，确认 FFmpeg 正确执行音视频合并
- **仅音频提取**：测试 `bestaudio` + `FFmpegAudio` postprocessor，确认输出 MP3/AAC 文件质量正常
- **断点重试机制**：模拟中断场景，验证 `--retries` 参数生效（默认重试 10 次）

---

## 三、FFmpeg 媒体处理验收

FFmpeg 是音视频转码与合并的核心，需单独验收其集成逻辑：

- **音视频合并**：检查下载后的 MKV/MP4 容器是否同时包含视频流和音频流（用 `ffprobe` 探测）
- **格式转换**：测试从 WebM → MP4 的转码流程，验证输出文件可正常播放
- **元数据写入**：确认标题、封面图、时长等元数据是否被正确嵌入输出文件
- **`--fixup` 策略**：验证文件修复策略（`detect_or_warn`）对损坏片段的处理行为
- **并发分片下载**：测试 `--concurrent-fragments` 参数，验证大文件分片下载后合并无损

---

## 四、错误处理与边界场景测试

| 场景 | 预期行为 | 检查方法 |
|------|----------|----------|
| 私有 / 已删除视频 | 输出明确错误信息，不崩溃 | 用失效 URL 测试 |
| 需要登录的视频 | 提示 "Sign in required"，引导使用 cookies | 测试会员内容 URL |
| 网络中断 | 自动重试，超限后优雅退出 | 模拟断网或限速 |
| DRM 保护视频 | 明确提示不支持，不假装成功 | 测试已知 DRM 内容 |
| 无效 URL 格式 | 参数校验层拦截，返回友好提示 | 传入随机字符串 |
| 磁盘空间不足 | 下载前预检或下载失败时清理临时文件 | 限制磁盘配额测试 |

---

## 五、Deno 集成验收

如果 Deno 负责调度或 API 层，需专项验收其与 Python 的交互：

- **进程通信**：验证 Deno 通过 `subprocess` / IPC 调用 Python 脚本时，stdout/stderr 传递无丢失
- **异步任务队列**：测试多个下载任务并发时，Deno 的任务管理是否正确排队与回调
- **接口契约**：验证 Deno 暴露的 API 返回的 JSON 结构（文件路径、状态码、错误信息）与约定一致

---

## 六、性能与稳定性验收

- **限速测试**：使用 `-r 500K` 限制下载速率，确认程序运行稳定不超时
- **长时运行**：连续运行 50+ 个视频的播放列表下载，监控内存泄漏和进程僵尸
- **并发安全**：同时触发 3~5 个下载任务，检查输出文件无互相覆盖
- **临时文件清理**：下载完成后确认 `.part`、`.ytdl` 等临时文件被正确删除

---

## 七、回归与自动化测试建议

完成手动验收后，建议建立以下自动化保障机制：

- 用 `pytest` 编写单元测试，mock `yt_dlp.YoutubeDL` 以隔离网络依赖
- 用 `subprocess` 调用 `ffprobe` 对输出文件做自动化媒体探测断言
- 集成 CI（如 GitHub Actions），在每次 push 时运行轻量级烟测（smoke test）
- 定期（如每周）对真实 YouTube URL 运行端到端回归测试，因为 YouTube 平台结构变更可能随时破坏下载逻辑
